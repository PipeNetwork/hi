# Migrating to hi 0.2

hi 0.2 was an intentional API, CLI, evaluation-task, and report-schema break for
the core agent. Current workspace crates that follow that break ship as
**0.3.x** (see root `Cargo.toml`). GPU and local-inference surfaces
(`hi-mlx`, `hi-cuda`, `hi-local`, `hi-local-core`, and `hi-gguf`) remain on a
separate version line and are outside the core agent release.

## Tool and turn APIs

`hi_tools::ToolOutput` has been replaced by `hi_tools::ToolOutcome`. Callers
must inspect its typed `status`, `process`, and `effects` fields instead of
inferring success from rendered text. `ToolEffects` distinguishes an attempted
mutation from one that was applied and lists the exact file changes, including
before/after digests, lengths, modes, and create/modify/delete kind.

`Agent::run_turn` now returns `anyhow::Result<TurnOutcome>`. A successful Rust
return value means that the turn driver completed normally; it does not by
itself mean the requested work passed its completion contract. Inspect
`TurnOutcome.status`, `verification`, and `review`. The outcome also records the
stop reason, changed files, verified workspace revision, and effective model
route.

`Agent::new` and `Agent::resume` also return `anyhow::Result<Agent>` in 0.2.
Runtime initialization validates the explicit workspace/state roots and
verification stages instead of panicking or falling back to process-global
state. Tool execution should be routed through the agent-owned
`WorkspaceRuntime`; the 0.1 cwd/global tool, cache, background, and LSP entry
points were removed.

The new status enums are:

- `ToolStatus`: `Succeeded`, `Failed`, `Denied`, `TimedOut`, `Cancelled`.
- `TurnStatus`: `Completed`, `Incomplete`, `Blocked`, `Cancelled`, `Failed`.
- `VerificationStatus`: `Passed`, `NotApplicable`, `Unverified`, `Failed`,
  `InfrastructureError`.
- `ReviewStatus`: `NotRequired`, `Passed`, `Objected`, `Unavailable`.

## Verification defaults and CLI

Deterministic verification is automatic by default. hi detects a staged
pipeline from the workspace and allows an initial check plus two repair/check
cycles. A pass applies only to the exact workspace revision that was checked;
any later mutation invalidates it.

The 0.1 flags `--auto-verify`, `--max-verify`, and `--minimal-tools` were
removed. Use:

```text
--verify CMD              replace detection with this stage (repeatable)
--no-verify               disable deterministic verification
--allow-unverified        permit an unverified mutation to exit successfully
--allow-no-checkpoint     retain YOLO fallback with --confirm-edits
--max-verify-repairs N    repair/check cycles after the initial check
--review risk|always|off
--lsp auto|on|off
--tool-set dynamic|minimal|full
```

When `--max-steps` is omitted, the automatic caps are 80 model calls for a
clearly read-only turn, 120 for recognized implementation work, and 200 for a
general or ambiguous turn. Ambiguous turns still receive mutation safeguards.

One-shot exit codes are now stable:

| Code | Meaning |
|---:|---|
| 0 | Completed and passed/not-applicable, or explicitly allowed unverified work |
| 1 | Incomplete, blocked, verification failed, or an unverified mutation |
| 2 | Configuration or usage error |
| 3 | Unrecovered provider, process-runner, or internal infrastructure failure |
| 130 | User cancellation |

## Project quality configuration

Repository settings live in `.hi/config.toml`. CLI values take precedence,
then project configuration, then automatic detection.

```toml
[quality]
verification = "auto" # or "disabled"
max_verify_repairs = 2
review = "risk"       # risk, always, off
lsp = "auto"          # auto, on, off
tool_set = "dynamic"  # dynamic, minimal, full
context_exclusions = ["generated/**"]

# To replace detection, change verification to "explicit" and add:
# stages = ["cargo check --workspace", "cargo test --workspace"]
```

## Reports and sessions

New reports use schema version 2 only. They contain the typed turn outcome,
configured verification stages separately from actual typed stage executions,
review result, typed tool statuses, actual model
route, turn/session usage, and exact changes. Consumers must not rely on 0.1
inferred fields. hi can still read historical schema-v1 reports and legacy
session checkpoint strings for migration, but never emits them. For example,
read session token totals from `usage.session.total_tokens`; the 0.1
top-level `total_tokens` field is not emitted.

## Evaluation task schema

Task schema v2 separates feedback visible during an attempt from the immutable
final oracle. A task declares its prompt, allowed-change globs, optional visible
feedback command, final oracle bundle/command, and independent timeouts for
candidate execution, visible feedback, and the final oracle. The evaluator
captures the oracle before starting a candidate and injects it only into a fresh
post-attempt verification copy. Candidate-side test edits therefore cannot alter
the final score. Defaults are three trials, a 900-second candidate timeout, and
a 120-second final-oracle timeout.

Evaluation artifacts retain every candidate, including temperature/seed,
actual route, outcome, patch, trace, checks, tokens, known cost, and wall time.
Candidate pass rate and `solve@N` are reported separately; standard `pass@k`
is emitted only for exchangeable repeated samples.
